# 给 VS Code Agent 接入 MCP

> 目标：只为一个明确需求添加 MCP 工具，并先用低风险动作确认它的权限范围。

## 直接照做

1. 先写下缺少的能力，例如“读取工单”或“查询测试数据库”。
2. 选择该服务的官方或可信 MCP 服务器，查看它会启动什么命令、访问哪些数据。
3. 按服务器说明把配置加入 `.vscode/mcp.json`，Secret 通过环境变量或安全存储传入。

> [此处应有：图 01——`.vscode/mcp.json` 的示例结构；框出服务器名称、command、args 和环境变量引用；所有真实 Token 与内部地址隐藏]

4. 在 VS Code 中检查服务器暴露的工具列表，只启用当前任务需要的部分。
5. 第一次调用选择只读、范围小的动作，例如列出一条测试记录。
6. 确认结果和日志正常后，再逐步开放写入类工具。

> [此处应有：图 02——VS Code MCP 服务器状态与工具列表；框出已连接状态、只读测试工具和写入工具的审批入口]

## 怎么确认成功

- MCP 服务器能启动，VS Code 没有连接错误。
- Agent 只能看到预期工具，工具参数与配置一致。
- 测试调用访问的是预期环境，没有把 Key 写进 `.vscode/mcp.json`。
- 写入动作仍有清楚的审批或权限边界。

## Windows 上多看一眼

VS Code 当前不为本地 MCP 服务器提供 Windows sandbox。本地服务器能做什么，取决于启动它的账号权限，所以配置里的命令、参数、环境变量和工作目录值得逐项看一遍。

这不是要把每个 MCP 当成神秘生物，只是先确认它的牙口有多大。

## 什么时候用

当 Agent 需要 VS Code 内置工具之外的能力，例如浏览器自动化、数据库、工单或内部服务时，再接入 MCP。已有文件、终端或扩展能完成的任务，不必为了“工具很多看起来很热闹”再加一层。
